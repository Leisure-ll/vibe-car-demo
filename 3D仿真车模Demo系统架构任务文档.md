# 3D 仿真车模 Demo 系统架构任务文档

## 1. 文档目的

本文档用于从高级架构工程师视角，对 3D 仿真车模 Demo 进行系统架构推演。

本阶段不编写具体代码，重点明确：

- 项目应拆分为哪些模块；
- 用户操作、语音输入、车辆状态和 3D 画面之间的数据如何流转；
- 选择什么技术方案更适合本次 Vibe Coding 面试作业；
- 每个技术选择背后的理由；
- 后续开发任务应如何分层推进。

## 2. 架构设计目标

本项目不是量产车机系统，而是一个面向面试展示的网页 Demo。因此架构设计应优先满足以下目标：

- 可演示：核心功能打开即用，适合现场展示；
- 可控：功能边界清晰，避免范围膨胀；
- 可解释：每个模块和技术选择都能向面试官讲清楚；
- 可扩展：后续可以继续增加车门、灯光、天窗等控车功能；
- 可维护：点击控车、语音控车、3D 动画不应互相写死；
- AI 友好：模块拆分清晰，便于 AI 编程工具分任务生成、调试和优化。

## 3. 总体架构概览

系统可以理解为一个运行在浏览器中的前端单页应用，内部由 3D 车模展示层、交互控制层、语音识别层、车辆状态层和 UI 反馈层共同组成。

整体架构如下：

```text
用户
  |
  | 鼠标拖动 / 点击 / 语音输入
  v
交互输入层
  |
  | 标准化为控制意图
  v
控制指令层
  |
  | 更新车辆状态
  v
车辆状态层
  |
  | 驱动视觉变化 / UI 变化
  v
3D 车模展示层 + 页面反馈层
```

核心思想：

点击控制和语音控制不直接操作 3D 模型，而是先转换成统一的“车辆控制指令”，再由车辆状态层更新状态，最后驱动 3D 车模和 UI 同步变化。

这样做的好处是：

- 点击和语音两种入口可以复用同一套控制逻辑；
- 后续新增车门、灯光、后备箱等功能时，不需要重写整体结构；
- 面试时可以清楚说明系统的数据流和模块分工。

## 4. 系统模块划分

### 4.1 页面应用壳模块

职责：

- 负责页面整体布局；
- 承载 3D 车模区域、控制面板、语音入口、状态提示；
- 管理页面级初始化流程；
- 组织各模块之间的组合关系。

包含内容：

- 顶部或侧边状态区域；
- 中央 3D 车模展示区；
- 车窗控制按钮；
- 语音控制按钮；
- 操作反馈提示区。

设计原则：

- 页面第一屏应直接展示车模，不做营销式落地页；
- 功能入口要清楚，适合面试官快速试用；
- UI 不应遮挡 3D 车模的主要观察区域。

### 4.2 3D 车模展示模块

职责：

- 加载并渲染 3D 汽车模型；
- 设置摄像机、灯光、材质、环境效果；
- 支持用户拖动旋转查看车辆；
- 根据车辆状态变化播放对应动画或切换部件状态。

核心能力：

- 车模展示；
- 轨道旋转；
- 车窗状态可视化；
- 基础动画；
- 部件命中检测。

车窗表现建议：

- 最小版本可以通过车窗位置、透明度、开合动画或位移变化表现“打开 / 关闭”；
- 如果模型结构允许，优先让车窗玻璃产生上下移动效果；
- 如果模型不支持独立车窗部件，可以使用 UI 状态和局部高亮作为兜底反馈。

### 4.3 车辆状态管理模块

职责：

- 保存车辆当前状态；
- 接收标准化控制指令；
- 根据指令更新状态；
- 通知 3D 车模模块和 UI 模块刷新。

建议管理的状态：

```text
车辆状态
  - 左前车窗：打开 / 关闭
  - 右前车窗：打开 / 关闭
  - 左后车窗：打开 / 关闭
  - 右后车窗：打开 / 关闭
  - 当前选中部件
  - 最近一次控制指令
  - 最近一次语音识别文本
  - 最近一次执行结果
```

最小版本可简化为：

```text
车辆状态
  - 车窗：打开 / 关闭
  - 最近一次控制来源：点击 / 语音
  - 最近一次执行结果
```

设计原则：

- 车辆状态是系统的单一事实来源；
- 点击、语音和 UI 显示都围绕车辆状态运转；
- 不应让不同模块各自维护一套车窗状态，避免状态错乱。

### 4.4 点击控车模块

职责：

- 处理用户点击车窗、按钮或车模部件的行为；
- 将点击行为转换为标准控制指令；
- 调用车辆状态管理模块完成状态更新。

输入来源：

- 点击 3D 车窗部件；
- 点击页面上的“打开车窗 / 关闭车窗”按钮；
- 点击车窗状态切换控件。

输出结果：

```text
控制指令
  - 指令类型：打开 / 关闭 / 切换
  - 控制对象：车窗 / 左前车窗 / 全部车窗
  - 指令来源：点击
```

设计原则：

- 优先保证按钮控制稳定可用；
- 如果 3D 部件点击识别复杂，可将其作为增强功能；
- 点击操作和语音操作应复用同一套指令处理逻辑。

### 4.5 语音控车模块

职责：

- 获取浏览器麦克风权限；
- 识别用户语音；
- 将识别文本解析为车辆控制指令；
- 将指令发送给车辆状态管理模块；
- 对失败场景给出提示。

语音输入示例：

- 打开车窗；
- 关闭车窗；
- 打开左前车窗；
- 关闭全部车窗。

语音处理流程：

```text
用户点击语音按钮
  -> 浏览器请求麦克风权限
  -> 用户说出指令
  -> 浏览器返回识别文本
  -> 指令解析模块匹配意图
  -> 生成标准控制指令
  -> 更新车辆状态
  -> 3D 车模和 UI 同步变化
```

设计原则：

- 本 Demo 不需要复杂自然语言理解；
- 采用固定指令词匹配即可满足面试展示；
- 必须考虑浏览器不支持语音识别的兜底提示；
- 语音失败不应影响点击控车。

### 4.6 指令解析模块

职责：

- 将不同输入来源转换为统一的车辆控制指令；
- 屏蔽点击和语音的差异；
- 对无法识别的输入返回失败结果。

标准控制指令结构：

```text
控制指令
  - action：open / close / toggle
  - target：window / frontLeftWindow / allWindows
  - source：click / voice
  - rawText：原始语音文本，可选
```

示例：

```text
用户说：“打开车窗”
  -> action = open
  -> target = window
  -> source = voice
```

```text
用户点击“关闭车窗”
  -> action = close
  -> target = window
  -> source = click
```

设计原则：

- 输入多样，指令统一；
- 指令解析层不直接改 3D 模型；
- 后续扩展车门、车灯时，只需要新增指令映射规则。

### 4.7 UI 控制与反馈模块

职责：

- 展示当前车辆状态；
- 展示语音识别结果；
- 展示操作成功或失败提示；
- 提供稳定的控车按钮；
- 帮助面试官理解当前 Demo 正在发生什么。

建议展示内容：

- 当前车窗状态：已打开 / 已关闭；
- 最近一次控制方式：点击 / 语音；
- 最近一次语音识别文本；
- 最近一次执行结果；
- 语音能力不可用时的提示。

设计原则：

- UI 是 3D 反馈的补充，不应喧宾夺主；
- 关键状态必须能被用户看见；
- 面试演示时，即使 3D 动画不明显，UI 也能证明控制逻辑有效。

### 4.8 资产管理模块

职责：

- 管理 3D 模型、贴图、环境图、图标等静态资源；
- 控制资源大小，避免页面加载过慢；
- 为后续替换更高质量车模留出空间。

资源类型：

- 3D 汽车模型；
- 车漆、玻璃、轮胎等材质贴图；
- 环境反射图；
- UI 图标；
- 演示截图或录屏。

设计原则：

- 资源质量直接影响“高还原”印象；
- 优先使用可在网页中稳定加载的模型格式；
- 资源不应过大，以免影响面试现场加载。

### 4.9 核心攻克模块

结合本项目需求，以下模块是最需要重点攻克的部分。这些模块直接决定 Demo 是否像新能源汽车中控大屏、是否具备真实交互感，以及是否能在面试中形成完整的技术表达。

#### 4.9.1 3D 车模展示模块

优先级：最高。

核心原因：

- 这是面试官打开 Demo 后的第一视觉印象；
- 题目明确要求“车模高还原”，该模块直接对应这一要求；
- 如果车模粗糙，即使交互功能可用，也会削弱项目完成度；
- 新能源汽车中控大屏的核心体验就是高质量 3D 车模展示。

需要攻克的问题：

- 车模比例是否接近真实汽车；
- 车身、车窗、轮胎、车灯等关键部件是否清晰；
- 玻璃、车漆、轮胎等材质是否有基本质感；
- 光照和相机角度是否能营造车机中控屏风格；
- 模型文件大小是否适合网页加载。

验收表现：

- 页面打开后，用户能一眼看出这是新能源汽车风格的 3D 车模；
- 车模不是静态图片，也不是简单几何体；
- 模型显示完整，没有明显穿模、遮挡或比例异常。

#### 4.9.2 3D 交互与拖动旋转模块

优先级：最高。

核心原因：

- “可拖动旋转”是题目明确要求的基础交互；
- 该能力决定 Demo 是否从“展示页”变成“可交互车机体验”；
- 拖动旋转是用户验证 3D 真实性的第一动作。

需要攻克的问题：

- 鼠标或触控拖动是否顺滑；
- 旋转方向是否符合用户直觉；
- 相机距离、旋转角度是否合理；
- 拖动旋转是否会误触发控车点击；
- 旋转后车窗点击或按钮控制是否仍然正常。

验收表现：

- 用户按住车模区域左右拖动时，车模能稳定旋转；
- 旋转过程没有明显卡顿、跳动或画面丢失；
- 拖动交互与控车交互互不干扰。

#### 4.9.3 车辆状态管理与统一指令模块

优先级：最高。

核心原因：

- 点击控车和语音控车都需要落到同一套车辆状态；
- 它决定系统是否只是“临时写死效果”，还是具备清晰架构；
- 这是面试中最能体现工程设计能力的部分。

需要攻克的问题：

- 如何定义车窗打开、关闭等车辆状态；
- 如何让点击输入和语音输入生成统一控制指令；
- 如何避免 UI 状态、3D 模型状态和内部状态不一致；
- 如何为后续车门、灯光、天窗等功能扩展预留结构。

验收表现：

- 点击“打开车窗”和语音说“打开车窗”走同一套业务逻辑；
- 状态变化后，3D 车模和页面文字同步更新；
- 连续操作不会出现状态错乱；
- 新增控制对象时，不需要推翻现有结构。

#### 4.9.4 点击控车模块

优先级：高。

核心原因：

- “可点击控车”是题目明确要求；
- 它是最稳定、最适合面试现场演示的控车路径；
- 即使语音识别受浏览器影响，点击控车也必须可靠。

需要攻克的问题：

- 点击按钮控制与点击 3D 部件控制如何取舍；
- 如果实现 3D 部件点击，如何准确识别车窗区域；
- 如何给用户明确的点击反馈；
- 如何避免重复点击导致动画或状态混乱。

验收表现：

- 用户能通过点击打开和关闭车窗；
- 页面能明确显示当前车窗状态；
- 点击操作响应及时；
- 点击控车链路在不同浏览器中稳定可用。

#### 4.9.5 语音控车模块

优先级：高。

核心原因：

- “可语音控车”是题目的亮点功能；
- 它能体现 Demo 的智能化和新能源车机体验；
- 它也是展示 AI 协作和需求理解的加分点。

需要攻克的问题：

- 浏览器是否支持语音识别；
- 麦克风权限如何提示；
- 语音识别结果如何转换为控制指令；
- 无法识别或不支持的指令如何反馈；
- 语音能力失败时如何不影响主流程演示。

验收表现：

- 用户说“打开车窗”后，车窗状态变为打开；
- 用户说“关闭车窗”后，车窗状态变为关闭；
- 页面显示识别文本和执行结果；
- 不支持语音时页面有提示，不发生崩溃。

#### 4.9.6 车窗动画与状态反馈模块

优先级：高。

核心原因：

- 控车动作必须让用户“看见变化”；
- 如果只有按钮文字变化，车机沉浸感不足；
- 车窗开关是本 Demo 最小闭环的核心视觉反馈。

需要攻克的问题：

- 模型车窗是否可以独立控制；
- 车窗动画使用位移、透明度还是局部高亮；
- 动画时间是否自然；
- UI 状态和 3D 动画是否同步；
- 动画失败时是否有文字反馈兜底。

验收表现：

- 执行打开或关闭车窗后，用户能明显感知状态变化；
- 动画与状态文案一致；
- 不会出现车窗显示打开但状态写着关闭的情况。

#### 4.9.7 演示稳定性与中间产物模块

优先级：中高。

核心原因：

- 这是面试作业，不只是开发功能；
- 交付物明确要求“有价值的中间产物”；
- 稳定演示和过程材料会影响 AI 工具使用程度、AI 交互协同程度等评分。

需要攻克的问题：

- Demo 是否能在面试环境稳定启动；
- 是否准备了需求文档、架构文档、运行说明和 AI 协作记录；
- 是否有清晰的演示路径；
- 出现语音不可用等异常时，是否有替代演示方案。

验收表现：

- 面试官能按说明打开 Demo；
- 候选人能按固定路径完成拖动、点击控车和语音控车演示；
- 文档能说明从需求理解到架构设计再到实现推进的过程。

#### 4.9.8 核心攻克优先级排序

建议按以下顺序投入精力：

1. 3D 车模展示模块；
2. 3D 交互与拖动旋转模块；
3. 车辆状态管理与统一指令模块；
4. 点击控车模块；
5. 车窗动画与状态反馈模块；
6. 语音控车模块；
7. 演示稳定性与中间产物模块。

排序理由：

- 先保证“看起来像”和“能拖动”，建立 Demo 的第一印象；
- 再建立统一状态和指令模型，避免后续点击、语音各写一套；
- 然后完成点击控车，这是最稳定的主链路；
- 接着优化车窗反馈，让控车效果可见；
- 最后接入语音控车，并准备兼容性兜底和面试材料。

## 5. 系统数据流转路径

### 5.1 页面初始化数据流

```text
用户打开网页
  -> 页面应用壳初始化
  -> 创建默认车辆状态
  -> 初始化 3D 场景
  -> 加载车模资源
  -> 初始化控制面板
  -> 初始化语音能力检测
  -> 页面进入可交互状态
```

关键点：

- 默认车辆状态应明确，例如车窗默认为关闭；
- 即使语音能力不可用，页面仍应允许点击控车；
- 车模加载过程中应有合理加载状态。

### 5.2 拖动旋转数据流

```text
用户按住并拖动车模区域
  -> 3D 交互控制器接收鼠标或触控事件
  -> 更新相机角度或车模旋转角度
  -> 渲染器刷新画面
  -> 用户看到车模旋转
```

关键点：

- 拖动旋转只影响观察视角；
- 不改变车辆业务状态；
- 不应触发车窗开关等控车动作。

### 5.3 点击控车数据流

```text
用户点击车窗按钮或车模车窗区域
  -> 点击控车模块识别点击对象
  -> 指令解析模块生成标准控制指令
  -> 车辆状态管理模块更新车窗状态
  -> 3D 车模展示模块播放车窗变化
  -> UI 反馈模块更新状态文案
```

示例：

```text
点击“打开车窗”
  -> 生成 open window 指令
  -> 车辆状态从 closed 变为 open
  -> 车窗视觉变为打开
  -> 页面显示“车窗已打开”
```

### 5.4 语音控车数据流

```text
用户点击语音按钮
  -> 语音控车模块开始识别
  -> 用户说出“打开车窗”
  -> 浏览器返回识别文本
  -> 指令解析模块匹配“打开 + 车窗”
  -> 生成 open window 指令
  -> 车辆状态管理模块更新状态
  -> 3D 车模和 UI 同步更新
```

异常路径：

```text
用户说出不支持的指令
  -> 语音模块返回识别文本
  -> 指令解析失败
  -> 车辆状态不变化
  -> UI 提示“不支持该指令”
```

```text
浏览器不支持语音识别
  -> 页面检测失败
  -> 禁用或提示语音入口
  -> 点击控车功能保持可用
```

### 5.5 状态同步数据流

```text
车辆状态变化
  -> 通知 3D 车模展示模块
  -> 通知 UI 反馈模块
  -> 记录最近一次操作结果
```

关键点：

- 所有可视化结果都应由车辆状态驱动；
- 避免点击按钮后只改 UI、不改模型；
- 避免语音控制后只改模型、不改状态文案。

## 6. 技术方案选择及理由

### 6.1 应用形态：网页单页应用

推荐方案：

- 使用网页单页应用作为 Demo 载体。

选择理由：

- 与题目“demo 形式为网页”一致；
- 面试官打开成本低；
- 不需要安装 App；
- 方便同时承载 3D 画面、按钮、状态面板和语音入口；
- 方便部署到本地或线上静态站点。

不优先选择：

- 原生 App：开发和安装成本高，不符合网页 Demo 要求；
- 小程序：平台限制较多，3D 和语音能力不如浏览器方案直观；
- 视频演示：无法证明真实交互能力。

### 6.2 前端框架：React

推荐方案：

- 使用 React 构建页面和交互状态。

选择理由：

- 组件化能力强，适合拆分 3D 车模、控制面板、语音按钮、状态反馈等模块；
- 状态驱动 UI 的模式清晰，适合表达车辆状态变化；
- 生态成熟，AI 编程工具对 React 代码生成和调试支持较好；
- 便于后续扩展更多控车功能。

可替代方案：

- Vue：同样可行，适合国内前端团队习惯；
- 原生 JavaScript：依赖少，但复杂交互下模块组织会更困难；
- Svelte：体验轻量，但面试中通用性和 AI 生成稳定性略弱。

本项目建议选择 React，主要是为了降低 AI 协作和后续维护成本。

### 6.3 3D 渲染方案：Three.js

推荐方案：

- 使用 Three.js 作为 3D 渲染基础。

选择理由：

- Three.js 是 Web 3D 场景中最成熟、最常见的方案之一；
- 支持模型加载、材质、灯光、相机、轨道控制、动画等能力；
- 能够实现接近车机大屏的 3D 车模展示；
- 生态丰富，遇到问题容易检索和让 AI 辅助排查；
- 适合浏览器运行，不需要额外客户端。

可替代方案：

- Babylon.js：功能完整，适合大型 3D 应用，但本 Demo 使用 Three.js 更轻量、更常见；
- CSS 3D：无法满足高还原车模要求；
- 纯 Canvas 2D：无法自然支持 3D 旋转和模型展示。

### 6.4 React 与 Three.js 集成方案：React Three Fiber

推荐方案：

- 使用 React Three Fiber 管理 Three.js 场景。

选择理由：

- 可以用 React 组件方式组织 3D 场景；
- 与 React 状态管理结合自然；
- 便于将车辆状态传递给 3D 车窗动画；
- AI 工具对这种模式支持较好；
- 后续增加车灯、车门、后备箱等部件时，结构更清晰。

可替代方案：

- 直接在 React 中手写 Three.js 初始化逻辑：控制力强，但代码更底层，AI 生成和维护成本更高；
- 使用现成 3D 低代码平台：可能快速，但可解释性和代码可控性较弱。

### 6.5 3D 模型格式：glTF / GLB

推荐方案：

- 使用 glTF 或 GLB 格式作为 3D 车模资源。

选择理由：

- glTF / GLB 是 Web 3D 常用模型格式；
- 支持模型、材质、贴图和动画；
- Three.js 生态对该格式支持成熟；
- 单文件 GLB 更适合 Demo 分发；
- 后续可以替换更高质量模型而不改变主架构。

注意事项：

- 模型文件不宜过大；
- 如果需要单独控制车窗，最好选择车窗部件可独立识别的模型；
- 如果模型无法拆分车窗，需要通过视觉效果或 UI 反馈兜底。

### 6.6 车控状态管理方案：轻量前端状态管理

推荐方案：

- MVP 阶段使用 React 内置状态或轻量状态管理方案。

选择理由：

- 当前 Demo 状态规模小，核心只有车窗开关、语音结果、操作反馈；
- 不需要引入复杂全局状态框架；
- 状态集中在车辆状态模块即可满足需求；
- 降低代码复杂度，方便面试讲解。

可升级方向：

- 当功能扩展到车门、灯光、空调、座椅、后备箱等多个域时，可引入 Zustand 等轻量状态库；
- 不建议一开始引入 Redux，除非后续状态复杂度明显提升。

### 6.7 语音识别方案：浏览器 Web Speech API

推荐方案：

- 使用浏览器 Web Speech API 实现语音识别。

选择理由：

- 不需要后端服务；
- 不需要接入真实语音云服务；
- 实现成本低，适合面试 Demo；
- 可以直接完成“打开车窗”“关闭车窗”等基础指令识别；
- 更能突出网页 Demo 的轻量可运行特点。

限制：

- 不同浏览器支持程度不同；
- 可能需要浏览器麦克风权限；
- 识别效果受系统环境影响；
- 某些浏览器可能不可用。

兜底方案：

- 页面检测不支持时显示提示；
- 保留点击控车作为稳定演示路径；
- 可提供文本指令输入作为可选兜底，但不作为首要功能。

### 6.8 指令理解方案：规则匹配

推荐方案：

- 使用固定规则匹配语音文本中的关键词。

示例：

```text
包含“打开”且包含“车窗” -> 打开车窗
包含“关闭”且包含“车窗” -> 关闭车窗
包含“左前”且包含“车窗” -> 控制左前车窗
```

选择理由：

- 本 Demo 指令范围有限；
- 规则匹配稳定、可解释、容易调试；
- 不需要引入大模型接口；
- 面试现场网络不稳定时风险更低；
- 更符合“模拟控车”的 MVP 范围。

不优先选择：

- 接入在线大模型做自然语言理解：能力强，但依赖网络和 API Key，面试现场不稳定；
- 复杂 NLP 框架：对本项目过重。

### 6.9 构建工具：Vite

推荐方案：

- 使用 Vite 作为前端开发和构建工具。

选择理由：

- 启动快；
- 配置简单；
- 与 React、Three.js 生态适配好；
- 适合快速构建面试 Demo；
- 生成的静态资源便于部署和交付。

可替代方案：

- Next.js：适合完整 Web 应用，但本项目不需要服务端渲染和复杂路由；
- Webpack：能力强，但配置成本更高；
- 静态 HTML：简单，但大型 3D 交互和状态管理会变得混乱。

### 6.10 样式方案：CSS Modules 或普通 CSS

推荐方案：

- 使用普通 CSS 或 CSS Modules 管理样式。

选择理由：

- Demo 样式规模有限；
- 容易控制车机风格界面；
- 不引入过多 UI 框架，避免页面看起来像普通后台系统；
- 方便 AI 工具直接生成和调整视觉效果。

不优先选择：

- 大型组件库：容易导致视觉风格偏管理后台，不像车机中控；
- 复杂 CSS-in-JS：对本 Demo 必要性不高。

## 7. 推荐系统分层

建议将系统逻辑按以下层次理解：

```text
展示层
  - 页面布局
  - 控制按钮
  - 状态提示
  - 3D 车模画面

交互层
  - 鼠标拖动
  - 车窗点击
  - 语音按钮

指令层
  - 点击事件转指令
  - 语音文本转指令
  - 非法指令处理

状态层
  - 车辆状态
  - 操作结果
  - 语音识别结果

渲染层
  - 3D 模型加载
  - 材质灯光
  - 车窗动画
  - 画面刷新
```

该分层能保证功能扩展时结构稳定。

## 9. 架构验收标准

### 9.1 模块边界验收

- 3D 展示逻辑与车辆状态逻辑分离；
- 点击控车和语音控车复用同一套控制指令；
- UI 状态展示由车辆状态驱动；
- 语音识别失败不影响点击控车；
- 后续新增车门、车灯功能时，不需要推翻整体结构。

### 9.2 数据流验收

- 用户点击后能形成标准控制指令；
- 用户语音输入后能形成标准控制指令；
- 控制指令能更新车辆状态；
- 车辆状态变化能同步反映到 3D 车模和 UI；
- 异常指令不会导致系统状态错乱。

### 9.3 技术方案验收

- 技术方案符合网页 Demo 要求；
- 3D 渲染方案能支持真实车模和旋转交互；
- 语音方案能支持基础语音控车；
- 技术栈复杂度适合面试作业；
- 面试时能够清楚解释每项技术选择的原因。

## 10. 架构风险与应对

### 10.1 模型不可控风险

风险：

- 下载或制作的车模可能无法单独控制车窗部件。

应对：

- 优先寻找部件可拆分的 GLB 模型；
- 如果车窗无法单独动画，使用透明度、局部高亮或 UI 状态作为兜底；
- MVP 阶段先保证控制闭环，再优化视觉动画。

### 10.2 语音兼容性风险

风险：

- Web Speech API 在部分浏览器中不可用。

应对：

- 推荐使用支持较好的浏览器进行演示；
- 页面启动时检测语音能力；
- 不支持时展示提示；
- 保留点击控车作为主演示链路。

### 10.3 3D 性能风险

风险：

- 模型过大、材质过多可能导致加载慢或卡顿。

应对：

- 控制模型文件大小；
- 减少不必要的高精度贴图；
- 优先保证主视角展示效果；
- 面试前准备本地可运行版本。

### 10.4 功能范围膨胀风险

风险：

- 过早扩展车门、灯光、空调、座椅等功能，导致核心功能不稳定。

应对：

- MVP 只做车窗控制闭环；
- 车门、车灯等作为加分项；
- 先稳定演示，再扩展功能。

## 11. 推荐 MVP 架构结论

本项目推荐采用以下 MVP 架构：

```text
Vite + React
  -> 页面应用与状态管理

React Three Fiber + Three.js
  -> 3D 车模展示、拖动旋转、车窗视觉变化

GLB 车模资源
  -> 网页友好的 3D 模型格式

Web Speech API
  -> 浏览器内语音识别

规则匹配
  -> 将语音文本转换为控车指令

统一车辆状态
  -> 驱动 3D 模型和 UI 反馈
```

该方案的核心价值是：

- 实现成本适中；
- 面试演示稳定；
- 技术路径容易解释；
- AI 编程工具容易协作；
- 后续扩展空间清晰。

## 12. 面试表达建议

在面试中可以这样解释架构：

“我没有让点击和语音直接操作 3D 模型，而是先把它们统一转换成车辆控制指令，再更新车辆状态，最后由状态驱动 3D 车模和 UI 变化。这样点击控车和语音控车复用同一套业务逻辑，后续要扩展车门、车灯、后备箱等功能，也只需要新增状态字段和指令映射，不需要重写整体系统。”

这个表达能体现：

- 对需求有抽象能力；
- 对前端交互架构有理解；
- 对 Demo 稳定性有考虑；
- 对未来扩展有预留；
- 不是简单让 AI 生成页面，而是在主动设计系统结构。
